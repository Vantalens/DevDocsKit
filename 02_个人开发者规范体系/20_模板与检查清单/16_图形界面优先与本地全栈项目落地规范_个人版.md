# 图形界面优先与本地全栈项目落地规范（个人版）

版本：v1.1.0
状态：生效
最后更新：2026-06-27

## 1. 目标
用于已有项目的持续演进，尤其适用于“静态前端 + 本地后端 + SQLite/文件数据 + 图形化管理后台”的个人项目。目标是让日常使用优先走图形界面，命令行只作为构建、排错和自动化验证工具。

## 2. 适用场景
- 项目已经存在，不能按新项目模板推倒重来。
- 使用者偏好图形界面，不希望日常操作依赖命令行。
- 项目同时包含前端、后端、本地数据库、构建产物和运行时数据。
- 需要 AI 协作进行审查、重构、修复和验证。

## 3. GUI 优先原则
- 日常入口必须明确到一个可双击或可固定到桌面的程序。
- 后台页面必须覆盖常用操作：文章、评论、设置、构建、状态检查。
- 命令行命令只保留为“维护命令”，文档中必须说明普通用户是否需要执行。
- 如果存在多个 exe，必须标注唯一日常入口，其余入口只能作为调试入口或不保留。

## 4. 本地全栈项目结构规则
每个项目必须记录：
- 前端构建入口与预览入口。
- 后端统一入口、调试入口和端口。
- 数据库存储位置、是否上传、备份策略。
- 运行时数据、缓存、日志、构建产物的 Git 上传边界。
- 图形界面 URL 与登录方式。

## 5. 数据与存储规则
- 用户生成数据优先进入后端数据库或受控数据目录。
- 如果前端仍依赖静态文件构建，后端写入数据库时必须明确是否同步文件。
- 数据库必须放入不可上传目录，除非明确是演示数据。
- 数据结构变更必须记录表名、字段、索引和迁移/回滚方式。
- 删除操作必须说明是否同步删除数据库记录、静态文件和缓存。

## 6. 安全与可靠性基线
后端至少满足：
- 管理接口必须鉴权。
- 写接口必须限制请求大小并进行服务端校验。
- 登录、评论、验证码等高风险接口必须限流。
- HTTP 服务必须设置读写超时和安全响应头。
- CORS 来源必须白名单化，部署域名通过配置扩展。
- 敏感配置、数据库、日志、构建产物不得上传 Git。

## 7. AI 协作执行规则

GUI、本地全栈和桌面工具必须同时验证自动化检查和真实用户路径。构建通过、接口健康检查通过或进程启动成功，不能替代真实 GUI 路径验证。

## 7. AI 协作执行规则
AI 接手已有项目时必须先做三件事：
1. 识别真实日常入口、调试入口和构建入口。
2. 识别运行时数据、数据库和不可上传文件。
3. 先跑最小验证，再做重构；重构后必须再次验证。
4. 识别本阶段可验收用户路径，并把验证证据写入阶段归档。

AI 修改已有项目时必须遵守：
- 不删除用户数据，不清空数据库，不重置 Git。
- 不把“缓存式同步”伪装成“数据库持久化”。
- 对 GUI 用户，最终说明必须给出“打开什么、点哪里、是否需要命令行”。
- P0/P1、安全或阻断当前 GUI 用户路径的问题必须当前阶段修复；非阻断问题进入 issue 和本地阶段报告。
- 代码审查结果必须转化为可执行修复，除非用户只要求报告。

## 8. 验收清单
- 是否只有一个清晰的日常 GUI 入口。
- 是否能通过浏览器完成主要前后端操作。
- 后端数据库是否实际生成并被真实接口读写。
- 构建命令是否通过，且不依赖未说明的缓存或本机状态。
- 是否完成真实 GUI 路径验证，并保留截图、录屏、日志或操作步骤证据。
- `.gitignore` 是否覆盖 exe、数据库、缓存、日志、构建产物和敏感配置。
- 最终说明是否包含普通用户路径和维护者路径。
