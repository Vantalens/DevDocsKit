# 实施计划与复现规范 IMPLEMENTATION_PLAN（个人版）

版本：v1.2.0
状态：生效
最后更新：2026-06-27

## 1. 目标
将需求拆解为可执行小任务，并保证问题可稳定复现与回归。

## 2. 任务拆分规则

进入实现前先冻结关键闭环：目标、范围、流程、技术栈、实施阶段、测试验收、回滚和 AI 执行边界。完整规格不要求一次写满，但每个小阶段开始前必须重新冻结本阶段范围和验收标准。

个人项目默认采用小阶段推进：一个小阶段对应一段可验收用户路径、一个可独立审查的文档阶段，或一个可验证修复集合。

- 每个任务 2 到 6 小时
- 一个任务一个目标
- 每个任务绑定 REQ 与 TEST
- 每个任务绑定一个主模块，跨模块任务必须说明影响范围
- 每个阶段完成后必须先代码审核，再进入下一阶段
- P0/P1、安全或阻断当前用户路径的问题在当前阶段内修复；非阻断问题进入 issue 和本地阶段报告后单独处理
- 关键代码提交前必须询问确认；文档提交按任务约定执行
- AI 不得提前实现未进入当前计划的复杂功能，可选优化登记为 issue

## 3. 任务条目必填
- TASK ID
- 对应 REQ ID
- 主模块
- codegraph 影响面梳理
- 输入条件
- 步骤
- 输出产物
- 验收方式
- 覆盖率目标与检查方式
- 代码审核与 issue 标签
- 本地阶段报告路径
- 小阶段 PR 边界
- 提交确认要求
- 回滚方式

## 4. 复现规范
### 4.1 环境复现
- OS 与运行时版本
- 依赖版本
- 环境变量名清单
- 启动命令顺序

### 4.2 问题复现
- 触发步骤
- 预期结果
- 实际结果
- 日志证据
- 影响范围

### 4.3 修复复现
- 根因
- 修复动作
- 回归验证
- 防复发规则

## 5. 成功判定

成功判定必须同时覆盖自动化结果和真实用户路径。Web、桌面、本地全栈或 GUI 项目不得把构建通过等同于用户路径通过。

- 在干净环境可重复执行至少 2 次
- 非作者可按文档在 30 分钟内复现
- 已使用 codegraph 或等价索引确认影响面
- 已完成阶段代码审核，问题已按标签提交 issue 或形成可提交 issue 内容，并同步写入本地阶段报告
- 检查覆盖率大于等于 80%，或明确记录阻断原因

## 6. 评审清单
- 是否可复现
- 是否有回滚
- 是否更新 progress 与 lessons
- 是否按模块拆分
- 是否完成 codegraph 梳理
- 是否完成代码审核和 issue 登记
- 是否避免 AI 过度超前
