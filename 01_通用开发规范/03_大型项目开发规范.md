# 大型项目开发规范

版本：v1.2.0  
状态：生效  
最后更新：2026-06-09

## 1. 目标

为大型项目建立可持续维护的工程规则，降低多人协作、多模块演进、长期交付中的失控风险。

大型项目的核心目标不是“把功能做完”，而是保证系统在需求持续变化、团队扩张、模块增多、运行时间变长之后，仍然可以被理解、被修改、被验证、被发布和被回滚。

## 2. 适用范围

满足任一条件时，应按本文执行：

- 项目预计维护周期超过 6 个月。
- 代码库超过 5 个核心模块，或存在多个前端、后端、任务、脚本、移动端、桌面端子系统。
- 参与者超过 3 人，或存在 AI 代理与人工协作的长期开发流程。
- 涉及账号、支付、权限、隐私、安全、数据迁移、生产运维等高风险能力。
- 项目需要灰度发布、版本兼容、回滚、监控告警或长期数据治理。

## 3. 基本原则

- 文档先行：需求、架构、接口、数据模型和验收标准必须先于关键代码变更。
- 边界清晰：模块必须有明确职责、输入输出、依赖方向和禁止访问范围。
- 小步交付：大型需求必须拆成可独立验证的里程碑，不允许一次性大合并。
- 可观测优先：核心链路必须具备日志、指标、追踪、告警和审计线索。
- 可回滚优先：任何生产变更都必须有回滚条件、回滚步骤和数据恢复策略。
- 单一事实来源：同一规则只能有一个主文档，其他文档通过引用使用。
- 审核先行：每个阶段完成后必须执行代码审核，并将问题按标签提交 issue 跟踪。
- 工具降本：优先使用 codegraph 或等价本地代码索引理解代码库，减少 token 消耗和管理成本。

## 4. 文档基线

大型项目至少维护以下文档：

1. `README.md`：项目入口、运行方式、目录说明、关键链接。
2. `PRD.md` 或需求目录：业务目标、范围、验收标准、非目标。
3. `APP_FLOW.md`：页面、路由、核心流程、异常流、状态机。
4. `TECH_STACK.md`：技术栈、版本、依赖策略、架构决策记录。
5. `ARCHITECTURE.md`：系统上下文、模块边界、依赖关系、部署拓扑。
6. `BACKEND_STRUCTURE.md`：数据模型、接口契约、权限、安全基线。
7. `IMPLEMENTATION_PLAN.md`：里程碑、任务拆分、验证方式、风险和回滚。
8. `TEST_PLAN.md`：测试分层、覆盖目标、关键用例、自动化策略。
9. `RELEASE_PLAN.md`：发布分级、灰度策略、回滚策略、发布检查。
10. `OPERATIONS.md`：监控、告警、日志、备份、故障处理。
11. `SECURITY.md`：权限模型、密钥管理、隐私数据、审计要求。
12. `progress.md` 与 `lessons.md`：进度、已知问题、经验沉淀。
13. `CODEGRAPH.md` 或等价记录：codegraph 初始化状态、索引范围、排除目录、模块图谱和常用查询方式。

## 5. 架构治理

### 5.1 模块边界

每个核心模块必须声明：

- 模块职责。
- 对外接口。
- 允许依赖的模块。
- 禁止依赖的模块。
- 数据读写边界。
- 模块负责人或维护责任人。
- 单元测试、集成测试和端到端测试入口。

禁止出现无边界的 `utils`、`common`、`shared` 膨胀层。通用模块必须说明抽象理由、使用者范围和兼容策略。

每个阶段实施前必须用 codegraph 或等价工具梳理相关模块、调用者、被调用者和潜在影响面，并把结论写入实施计划或 PR 描述。

### 5.2 依赖方向

- 业务层不得反向依赖表现层。
- 基础设施层不得携带业务规则。
- 跨模块调用优先通过公开接口、事件或服务契约完成。
- 禁止绕过模块边界直接访问对方私有数据表、内部类或内部目录。

### 5.3 架构决策记录

以下事项必须写入 ADR：

- 引入新框架、新数据库、新消息队列、新运行时。
- 改变模块边界、部署形态或数据所有权。
- 引入跨项目通用组件。
- 采用不可逆或高迁移成本方案。
- 影响安全、性能、成本或可维护性的关键选择。

ADR 至少包含：背景、决策、备选方案、取舍、影响范围、回滚策略。

## 6. 代码治理

### 6.1 代码所有权

- 每个核心模块必须有明确维护责任人。
- 关键目录应配置代码评审责任。
- 公共基础模块修改必须通知所有依赖方。
- 无负责人模块不得承载新核心能力。

### 6.2 分支与合并

- 主干必须保持可构建、可测试、可发布。
- 大功能使用短周期特性分支，定期同步主干。
- 禁止长期分支积累大量冲突后一次性合并。
- 每个合并请求必须关联需求、任务、测试证据和风险说明。

### 6.3 提交规则

提交应聚焦一个意图。禁止把格式化、重构、功能、依赖升级和测试修复混在同一个提交中。

强制要求：

- 一个 commit 只允许对应一个可验证、可回滚的原子任务。
- 大型需求必须先在 `IMPLEMENTATION_PLAN.md` 中拆分任务，再按任务逐个提交。
- 提交前必须查看 diff、确认暂存内容、执行最小验证并记录结果。
- 任何无法独立回滚的提交，必须在 PR 或变更记录中说明原因和回滚策略。
- 禁止为了减少提交数合并无关改动；提交数量多不是问题，提交意图混乱才是问题。

提交信息建议格式：

```text
type(scope): summary
```

常用 `type`：`feat`、`fix`、`refactor`、`test`、`docs`、`build`、`ci`、`chore`。

## 7. 数据治理

- 每张核心表必须有数据所有权说明。
- 字段新增、删除、改名必须有迁移脚本和回滚方案。
- 生产数据迁移必须先在等价环境演练。
- 高风险迁移必须分阶段执行：兼容字段、双写或灰度读、数据校验、切换、清理。
- 隐私数据必须定义采集目的、存储位置、访问权限、脱敏策略和保留周期。

## 8. 接口与契约

- API 必须有请求、响应、错误码、权限、幂等和限流说明。
- 跨服务接口必须有版本策略。
- 破坏性变更必须提供迁移窗口，不得直接删除旧契约。
- 关键接口必须具备契约测试或集成测试。
- 错误响应必须对用户可理解，对开发者可定位。

## 9. 测试门禁

大型项目至少执行四层测试：

1. 单元测试：验证模块内部业务规则。
2. 集成测试：验证模块之间、数据库、队列、缓存和外部服务交互。
3. 端到端测试：验证核心用户路径。
4. 非功能测试：验证性能、安全、兼容性、可用性和恢复能力。

发布前必须提供：

- 本次变更覆盖的测试清单。
- 检查覆盖率报告，强制大于等于 80%。
- 自动化测试结果。
- 手工验证证据。
- 未覆盖风险说明。
- 回归范围说明。

覆盖率不足 80%、安全高危问题、核心流程失败、回滚路径未验证时，不得进入发布。

## 9.1 代码审核与 Issue 跟踪

每个阶段结束后必须完成代码审核，审核内容覆盖正确性、健壮性、鲁棒性、安全性、性能、可维护性、可测试性和用户体验。

审核发现的问题必须提交 issue 或等价跟踪项，并带以下标签：

- 严重级别：`P0`、`P1`、`P2`、`P3`。
- 类型：`bug`、`security`、`performance`、`robustness`、`ux`、`test`、`docs`、`refactor`。
- 模块：对应核心模块或子系统。
- 阶段：需求、设计、实现、测试、发布或运维。

P0/P1、安全、核心流程、数据一致性和覆盖率阻断问题必须关闭或被负责人豁免后才能合并。

## 10. CI/CD 与发布

### 10.1 CI 基线

主分支至少包含：

- 格式检查。
- 静态检查。
- 单元测试。
- 构建验证。
- 关键集成测试。
- 依赖安全扫描。

### 10.2 发布分级

- P0：紧急修复，必须有最小变更、快速验证和回滚记录。
- P1：常规功能发布，必须完成测试门禁和发布检查。
- P2：架构、数据、权限或基础设施变更，必须有评审、演练、灰度和回滚。

### 10.3 灰度与回滚

- 高风险功能必须支持开关或灰度。
- 发布必须定义观测指标和停止条件。
- 回滚步骤必须在发布前写清楚。
- 涉及数据变更时，必须说明代码回滚和数据回滚是否独立可行。

### 10.4 语义化版本

- 发布版本采用 `MAJOR.MINOR.PATCH`。
- 不兼容公共 API、数据契约、配置契约、模块边界或用户工作流变更，升级 `MAJOR`。
- 向下兼容的新功能、新接口、新配置项、弃用声明或内部能力增强，升级 `MINOR`。
- 向下兼容的问题修复、安全补丁、文档修正或内部实现调整，升级 `PATCH`。
- 已发布版本不得原地修改；任何修正都必须发布新版本。
- 先行版本可使用 `-alpha`、`-beta`、`-rc`，构建信息可使用 `+build`。

## 11. 可观测性

核心链路必须具备：

- 结构化日志。
- 请求追踪 ID。
- 业务指标和技术指标。
- 错误率、延迟、吞吐量、资源使用告警。
- 用户影响范围判断方式。
- 故障复盘记录。

日志禁止记录真实密钥、密码、令牌、完整身份证件、完整银行卡、敏感隐私明文。

## 12. 安全与合规

- 权限默认最小化。
- 密钥必须通过环境变量、密钥管理服务或受控配置注入。
- 禁止将密钥、令牌、生产连接串提交到代码库。
- 所有外部输入必须校验。
- 管理后台、批量操作、导出能力必须有鉴权、审计和限流。
- 依赖升级必须检查破坏性变更和安全公告。
- 代码审核必须包含输入校验、权限边界、日志脱敏、依赖漏洞和敏感信息泄露检查。

## 13. AI 协作规则

AI 参与大型项目时必须遵守：

- 开始前先读取 README、架构文档、任务文档和相关模块规范。
- 开始前先使用 codegraph 或等价本地索引梳理目标模块、调用关系和影响面。
- 修改前明确影响范围和验证方式。
- 不擅自扩大重构范围。
- 不过度超前实现未来阶段功能；可选优化登记为 issue。
- 不删除未确认用途的代码、配置、脚本或数据文件。
- 不把无法验证的猜测写成已完成结论。
- 每次结束必须说明改了什么、如何验证、代码审核结果、issue 状态、覆盖率和剩余风险。

高风险任务必须拆分给 AI：

- 先读文档和代码。
- 再形成小范围方案。
- 再修改。
- 最后验证和记录。

## 14. 质量门禁

进入开发前：

- 需求范围明确。
- 架构影响明确。
- 数据影响明确。
- 验收标准明确。
- 回滚策略明确。

进入合并前：

- 代码通过评审。
- 测试通过。
- 检查覆盖率大于等于 80%。
- 审核问题已按标签提交 issue，阻断项已关闭或豁免。
- 文档同步。
- 监控和日志补齐。
- 风险有记录。

进入发布前：

- 发布计划完成。
- 回滚方案完成。
- 灰度策略完成。
- 关键路径验证完成。
- 负责人确认。

## 15. 常见反模式

- 一个巨大 PR 同时改需求、架构、样式、依赖和测试。
- 没有模块边界，所有代码都能互相调用。
- 只补单元测试，不验证真实集成路径。
- 只写成功流程，不写异常流和回滚。
- 文档和代码长期不同步。
- 引入新依赖但没有版本、风险和替代方案说明。
- 生产问题只修代码，不沉淀复盘和防复发规则。
- AI 跳过 codegraph 梳理直接大范围搜索和修改。
- 阶段结束只口头报告问题，不提交 issue 或等价跟踪项。
- 为了显得智能而提前实现未来功能、复杂配置或用户不需要的流程。
