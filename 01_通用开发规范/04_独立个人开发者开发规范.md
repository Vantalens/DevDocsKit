# 独立个人开发者开发规范（AI 协作精简版）

版本：v1.0.0  
状态：生效  
最后更新：2026-04-19

定位说明：本文件是个人开发速览规范；如需完整分文档体系，请使用「02_个人开发者规范体系」目录下的 00-15 全套文档。

## 1. 适用目标
本规范适用于单人开发者在 AI 协作场景下进行产品开发、迭代、发布与维护，目标是：
- 少文档但不失控
- 快迭代但可回滚
- 低成本但高复用
- 单人作战也可复现

## 2. 单人开发核心原则
- 文档先行，代码后置
- 小步快跑，每步可验证
- 任何改动可追踪到需求
- 每次发布可回滚
- 每次错误可沉淀复用经验

## 3. 最小文档栈（个人版）
个人开发只保留 6 份高价值文档：

1. PRD.md  
用途：定义做什么、不做什么、验收标准。

2. APP_FLOW.md  
用途：定义页面与流程，避免路由和交互混乱。

3. TECH_STACK.md  
用途：锁定依赖和版本，避免环境漂移。

4. IMPLEMENTATION_PLAN.md  
用途：把需求拆成可执行小任务。

5. progress.md  
用途：记录已完成、进行中、下一步、阻塞项。

6. lessons.md  
用途：记录错误模式和防复发规则。

说明：
- FRONTEND_GUIDELINES 与 BACKEND_STRUCTURE 可按项目复杂度选择性加入。
- 纯前端项目可暂不单独维护 BACKEND_STRUCTURE。

## 4. 单人开发工作流（强制）
### 4.1 开始前
- 更新 PRD：确认本轮目标与范围边界
- 更新 APP_FLOW：补齐成功流与失败流
- 更新 IMPLEMENTATION_PLAN：拆解到半天内可完成的小任务

### 4.2 开发中
- 一次只做一个任务
- 每完成一个任务即自测
- 每个任务完成后立即更新 progress
- 重要决策同步写入 TECH_STACK（如新增依赖、版本变更）

### 4.3 收尾时
- 更新 progress（Done/In Progress/Next/Blockers）
- 更新 lessons（本轮错误与纠正规则）
- 进行发布前检查清单

## 5. 任务拆分规范（个人版）
每个任务必须包含：
- TASK ID
- 对应需求 REQ ID
- 输入条件
- 实施步骤
- 预期结果
- 验收方法
- 回滚方法

任务粒度要求：
- 每个任务 2 到 6 小时可完成
- 一个任务只解决一个明确目标
- 禁止“超大任务”一次提交

## 6. AI 协作规范（个人版）
### 6.1 每次会话固定指令
在会话开始明确要求 AI：
- 先读取 PRD、APP_FLOW、IMPLEMENTATION_PLAN、progress、lessons
- 只执行当前任务，不越界改动
- 所有改动必须列出文件清单与验证方式

### 6.2 提示词结构
推荐固定结构：
- 背景：当前阶段与目标
- 输入：对应文档章节与任务 ID
- 输出：需要新增或修改的文件
- 约束：禁止新增依赖、禁止改动无关模块
- 验证：需要运行的测试或人工检查步骤

### 6.3 AI 输出验收
必须检查：
- 是否越界修改
- 是否引入未批准依赖
- 是否与 PRD 和 APP_FLOW 冲突
- 是否可在本地复现

## 7. 质量门禁（个人可执行版）
### 7.1 开发完成门禁
- 当前任务验收通过
- 追踪关系完整：REQ -> TASK -> TEST（至少手工测试）
- progress 已更新

### 7.2 发布门禁
- P0 问题为 0
- 核心流程手工回归通过
- 环境变量与密钥检查完成
- 回滚路径已演练

### 7.3 建议阈值
- 单元测试覆盖率大于等于 50%（逐步提升）
- 核心流程通过率 100%
- 关键接口 P95 小于等于 800ms（按项目可调整）

## 8. 安全与成本规范（单人必须执行）
### 8.1 安全底线
- 密钥只放环境变量，不写入仓库
- 前端禁止暴露后端密钥
- 所有输入默认不可信，服务端必须校验

### 8.2 成本控制
- 优先免费层与可替代方案
- 新增第三方服务前记录成本预估
- 每月做一次成本复盘：调用量、存储、带宽、AI Token 开销

## 9. 发布与回滚规范
### 9.1 发布步骤
1. 合并前执行最小回归
2. 打版本标签并记录变更摘要
3. 灰度或低峰发布
4. 观察核心指标与错误日志

### 9.2 回滚触发条件
- 关键流程不可用
- 错误率持续超阈值
- 数据一致性风险出现

### 9.3 回滚最小策略
- 保留上一稳定版本可快速切换
- 数据迁移脚本必须可回退或具备补偿策略

## 10. 复现规范（个人高优先）
每个线上问题必须记录：
- 触发条件
- 复现步骤
- 预期与实际结果
- 根因
- 修复动作
- 防复发规则（写入 lessons）

成功标准：
- 在干净环境能稳定复现与验证修复
- 未来自己 2 周后能看懂并复现

## 11. 个人节奏建议
- 每天开始：读 progress 与 lessons，选 1 到 3 个任务
- 每天结束：更新 progress，提交代码，记录 1 条 lessons
- 每周收尾：清理技术债，更新 TECH_STACK 与 PRD 的偏差项

## 13. 一页执行清单（每日）
- 今日目标是否来自 PRD 与 IMPLEMENTATION_PLAN
- 当前任务是否有清晰验收标准
- 是否仅改动必要文件
- 是否完成自测并更新 progress
- 是否记录 lessons

## 14. 禁止清单
- 未更新文档直接大规模编码
- 未测试直接发布
- 未准备回滚直接上线
- 将密钥提交到仓库
- 在同一轮中同时大改功能和架构
