## 1. 目标与适用范围

### 1.1 目标
建立统一、可执行、可审计的项目文档体系，降低 AI 与人工协作中的歧义、返工和架构漂移风险。

### 1.2 适用范围
- 适用于本项目所有新功能开发、重构、Bug 修复、上线发布与维护阶段。
- 适用于所有参与者：产品、设计、前端、后端、测试、运维、AI 代理。

### 1.3 核心原则
- 文档先于代码（Documentation First）
- 单一事实来源（Single Source of Truth）
- 小步迭代与持续校验（Small Batches + Validation）
- 规则可追踪、可更新、可复盘（Traceable + Evolvable）

## 2. 文档分层与最小集合

项目必须至少维护以下 8 份文档：

1. `PRD.md`：产品需求与范围边界
2. `APP_FLOW.md`：页面、路由、流程、状态流转
3. `TECH_STACK.md`：技术栈与依赖版本锁定
4. `FRONTEND_GUIDELINES.md`：设计系统与响应式规则
5. `BACKEND_STRUCTURE.md`：数据模型、接口契约、鉴权规则
6. `IMPLEMENTATION_PLAN.md`：分步骤实施计划
7. `AGENT_RULES.md`（或等价规则文件）：AI 执行约束
8. `progress.md`：会话进度、已完成/进行中/下一步

推荐新增：
- `CLAUDE.md`：Claude Code 或兼容 AI 代理的项目级执行入口，必须引用本规范、AGENT_RULES、progress 和测试/发布规则
- `lessons.md`：错误模式与纠正策略沉淀
- `README.md`：项目入口说明与本地运行方法

大型项目必须额外维护：
- `ARCHITECTURE.md`：系统上下文、模块边界、依赖方向、部署拓扑
- `TEST_PLAN.md`：测试分层、覆盖范围、关键用例、自动化策略
- `RELEASE_PLAN.md`：发布分级、灰度策略、回滚策略、发布检查
- `OPERATIONS.md`：日志、指标、告警、备份、故障处理
- `SECURITY.md`：权限模型、密钥管理、隐私数据、审计要求

## 3. 每份文档的编写标准

### 3.1 PRD.md
必须包含：
- 背景与业务目标
- 目标用户与核心场景
- 功能清单（In Scope）
- 明确不做（Out of Scope）
- 用户故事（As a..., I want..., so that...）
- 验收标准（可测试）
- 非功能要求（性能、安全、可用性、合规）

质量标准：
- 每个功能必须对应可验证结果
- 不允许使用“尽量/可能/适当”等模糊词而无量化定义

### 3.2 APP_FLOW.md
必须包含：
- 页面清单与路由映射
- 核心流程图（成功流、失败流）
- 每步触发条件、输入、输出
- 异常与回退策略
- 权限控制与重定向规则

质量标准：
- 任一流程必须可从入口追踪到终态
- 每个错误状态必须定义用户可见反馈

### 3.3 TECH_STACK.md
必须包含：
- 语言、框架、运行时版本
- 依赖库与固定版本（含关键插件）
- 构建、测试、部署工具链
- 新增依赖审批规则

质量标准：
- 禁止“最新版”表述，必须写确切版本
- 未列入文档的依赖不得直接引入

### 3.4 FRONTEND_GUIDELINES.md
必须包含：
- 设计令牌（颜色、字体、间距、圆角、阴影、动效时长）
- 布局规则（容器、栅格、断点）
- 组件规范（按钮、表单、卡片、弹窗等）
- 交互规则（加载、空态、错误态、禁用态）
- 可访问性（对比度、键盘可达、语义标签）

质量标准：
- 样式必须可复用，禁止大面积硬编码魔法值
- 明确移动优先（Mobile First）规则

### 3.5 BACKEND_STRUCTURE.md
必须包含：
- 数据库实体、字段、索引、关系
- API 契约（请求/响应、错误码、鉴权）
- 业务规则与边界条件
- 安全策略（输入校验、权限、限流、密钥管理）

质量标准：
- 每个接口必须有示例请求与响应
- 敏感数据处理策略必须显式声明

### 3.6 IMPLEMENTATION_PLAN.md
必须包含：
- 里程碑与阶段目标
- 原子任务拆分（建议 0.5-1 天可完成）
- 对应的小步提交计划：每个任务对应 1 个可回滚 commit，例外必须写明原因
- 每个任务的前置条件、输出产物、验收方式
- 风险与回滚方案

质量标准：
- 任务必须可独立验证，不允许“大而全”一步到位
- 禁止跨阶段跳步开发
- 禁止把多个独立意图塞进同一个提交

### 3.7 AGENT_RULES.md
必须包含：
- AI 可做与禁止做的行为清单
- 文件命名与目录约定
- 代码风格与测试策略
- 依赖引入与安全红线
- 执行顺序：先读哪些文档，再执行哪些任务

质量标准：
- 规则要可机器执行、可人工审查
- 每次发现重复错误必须更新本文件

### 3.8 progress.md
必须包含：
- 已完成（Done）
- 进行中（In Progress）
- 下一步（Next）
- 阻塞项（Blockers）
- 已知问题（Known Issues）

质量标准：
- 每次任务完成后立即更新
- 新会话开始前必须先阅读

## 4. 文档编写统一格式规范

### 4.1 结构规范
- 使用 Markdown 一级到三级标题，不超过三级
- 每份文档开头必须有元信息：版本、作者、更新时间、状态
- 长文档必须包含目录（TOC）

### 4.2 术语规范
- 统一术语表，避免同一概念多命名
- 缩写首次出现时写全称
- 英文技术名词保持一致，不混用别名

### 4.3 描述规范
- 使用“输入-处理-输出”表达逻辑
- 使用“条件-行为-结果”表达规则
- 需求与实现分离：PRD 不写具体代码方案

### 4.4 可验证性规范
- 每条关键规则要有验证方式
- 验收标准尽量量化（如响应时间、成功率、覆盖率）

## 5. 文档与开发流程的绑定规则

标准工作流：
1. 审问澄清：先补齐问题，再写文档
2. 冻结基线：确认并冻结本轮文档版本
3. 小步实施：按 `IMPLEMENTATION_PLAN.md` 执行
4. 小步提交：每完成一个可验证原子任务，先验证，再提交
5. 同步进度：每完成一项更新 `progress.md`
6. 纠错沉淀：错误修复后更新 `AGENT_RULES.md` 与 `lessons.md`
7. 发布验收：按发布检查清单走完整验证
8. 阶段审核：每个阶段完成后执行代码审核，将问题按标签提交 issue 或等价跟踪项

禁止行为：
- 未定义范围直接编码
- 未更新进度跨会话切换
- 未确认依赖直接新增库
- 未验证主流程直接发布
- 未验证即提交
- 把功能、重构、格式化、依赖升级、测试修复混入同一个 commit
- 阶段结束后跳过代码审核、问题登记或覆盖率门禁
- 未经需求和实施计划允许，提前实现复杂功能或未来阶段功能

## 6. 代码审核、Issue 与质量门禁

### 6.1 阶段代码审核

每个阶段开发完成后必须执行代码审核。审核范围至少覆盖：

- 正确性：需求、流程、边界条件和错误处理是否一致。
- 健壮性与鲁棒性：异常输入、部分失败、超时、重试、并发、资源释放是否处理。
- 安全性：鉴权、授权、输入校验、敏感信息、依赖安全和日志脱敏。
- 高性能：关键路径延迟、资源占用、重复计算、缓存、批量处理和性能退化风险。
- 可维护性：模块边界、命名、抽象层级、重复代码和可读性。
- 可测试性：测试入口、夹具、依赖隔离、回归用例和覆盖率。
- 用户体验：交互是否符合用户习惯，错误、加载、空态和恢复路径是否清晰。

### 6.2 Issue 标签化报告

审核问题必须提交 issue 或等价跟踪项，不得只保留在聊天记录中。每个问题至少包含：

- 标题：说明模块、问题类型和影响。
- 正文：现象、影响范围、复现步骤、证据、期望结果、建议修复方向。
- 标签：严重级别（`P0/P1/P2/P3`）、类型（`bug/security/performance/robustness/ux/test/docs/refactor`）、模块、阶段。
- 状态：`open`、`blocked`、`fixed`、`verified` 或 `wontfix`，其中 `wontfix` 必须有负责人说明。

P0/P1、安全、核心流程、数据一致性和覆盖率阻断问题必须在合并或发布前关闭，或形成明确的负责人豁免记录。

### 6.3 检查覆盖率

- 代码检查覆盖率强制要求大于等于 80%。
- 覆盖口径应同时包含自动化测试、静态检查、类型检查、lint、安全扫描、关键路径手工验证和代码审核覆盖模块。
- 核心域、安全边界、异常路径、回滚路径和高风险模块不得以平均覆盖率达标为由遗漏。
- 若项目暂无覆盖率工具，必须先定义可执行的替代检查清单，并在后续任务中补齐覆盖率统计。

## 7. 评审与验收标准（文档层）

### 7.1 文档评审清单
- 完整性：8 份核心文档是否齐全
- 一致性：文档间是否相互引用且无冲突
- 明确性：是否存在高歧义表述
- 可执行性：是否能直接指导实现
- 可测试性：是否有明确验收标准

### 7.2 质量评级
- A：可直接执行，低歧义，验收明确
- B：可执行，有少量待补充
- C：存在关键缺失，不可进入开发

进入开发门槛：
- PRD / APP_FLOW / TECH_STACK / IMPLEMENTATION_PLAN 至少达到 B
- AGENT_RULES 与 progress 文件必须存在并初始化

大型项目进入开发门槛：
- 已完成模块边界说明和架构决策记录
- 已确认数据影响、接口兼容策略和回滚策略
- 已明确测试分层、发布分级和观测指标
- 高风险变更已拆分为可独立验证的里程碑

## 8. 安全与合规写作红线

- 禁止在文档中记录真实密钥、密码、令牌
- 环境变量仅记录变量名与用途，不记录值
- 涉及隐私数据必须标注采集目的、存储位置、保留策略
- 外部服务接入必须记录权限范围与撤销方案

## 9. AI Skill 与工具能力准入

### 9.1 能力来源分级

AI skill、研究 skill、代码理解工具和代理工作流按以下优先级准入：

1. Codex 当前环境已支持的 skill、插件或 MCP 工具：优先使用，必须遵循其 `SKILL.md` 或工具说明。
2. 已在个人 GitHub star 中长期关注、且与当前任务直接相关的开发或研究 skill：可作为方法论参考或候选能力。
3. 其他外部 skill、脚本、模板或代理框架：只有在前两类不能覆盖任务时才允许评估。

严禁把 GitHub star 等同于自动采纳。star 只代表候选信号，不能替代安全审查、许可审查、维护状态审查和本地验证。

### 9.2 当前优先关注方向

结合已关注的开发与研究类项目，优先把下列能力沉淀为项目规则，而不是直接复制外部仓库内容：

- 代理工作流：任务分解、上下文窗口管理、记忆、编排、人工确认门禁。
- 代码理解：本地代码索引、知识图谱、调用关系、影响面分析，优先减少盲目全文搜索。
- AI 代码审核：证据优先、风险排序、可复现验证、少情绪化结论。
- 学术与科学研究：文献检索、研究问题拆解、实验记录、同行评审式修订。
- 工程化 skill 库：把可重复流程写成可执行 checklist、模板或脚本，避免只写抽象原则。
- 本地优先工具：代码、文档、OCR、知识库和代理运行优先保证本地可复现、可审计、可回滚。

### 9.3 引入或启用 Skill 的记录要求

任何项目若新增或启用 AI skill、研究 skill、代码理解工具、MCP 工具或代理插件，必须在 `TECH_STACK.md`、`AGENT_RULES.md` 或等价文档中记录：

- 名称、来源、版本或提交标识。
- 适用任务：开发、测试、代码审核、研究、文档、数据分析、部署或运维。
- 权限边界：可读写范围、是否需要网络、是否接触密钥、是否调用外部服务。
- 输入输出：读取哪些文件，生成哪些产物，是否会修改代码或文档。
- 验证方式：最小可运行样例、回归检查、人工复核点。
- 退出条件：何时停用、替换或回滚该能力。

### 9.4 执行约束

- 任务开始前必须先检查本地可用 skill，不能跳过已支持能力另起一套流程。
- 对研究类或外部知识类 skill，必须区分“资料检索结论”和“项目内已验证事实”。
- 对代码理解类 skill，输出只能作为定位和影响面分析依据，最终仍需通过测试、构建、审查或人工验收确认。
- 对能写文件、执行命令、访问网络或读取密钥的 skill，必须先说明权限、风险和最小执行范围。
- 每次因外部 skill 改变项目流程时，必须同步更新 `AGENT_RULES.md`、`progress.md`、`lessons.md` 或对应项目文档。
- Codex 或兼容 AI 代理每次开发前必须阅读项目入口开发文档，并优先用 codegraph 或等价本地代码索引梳理代码库影响面。

### 9.5 Codegraph 使用规则

- 项目库应使用 codegraph 建立本地代码知识图谱，梳理模块、符号、调用关系和影响面。
- 开发前优先通过 codegraph 回答“在哪里改、影响谁、谁调用它、模块边界是什么”。
- codegraph 用于降低 token 使用量和代码库管理难度，但不得替代测试、构建、安全扫描或人工审核。
- 新项目或未初始化项目应把 codegraph 初始化列入技术栈或实施计划，记录索引范围和排除目录。

## 10. 模块化开发与 UI 体验约束

### 10.1 模块化开发

- 开发必须按模块拆分，先定义模块职责、公开接口、依赖方向和数据边界。
- 每个任务必须绑定到一个主模块；跨模块任务必须说明契约变化、影响范围和回滚方式。
- 禁止无边界的 `utils`、`common`、`shared` 膨胀；公共抽象必须说明复用者和维护责任。
- 模块化不是文件拆分数量，而是职责清晰、依赖可控、验证可独立执行。

### 10.2 AI 开发节制

- AI 不得过度超前，不得在当前需求、实施计划或用户明确要求之外加入复杂功能。
- 对未来优化、架构升级、自动化增强和可选体验改进，优先登记 issue，而不是混入当前开发阶段。
- 若发现现有方案无法满足质量目标，应先回到文档更新和任务拆分，而不是扩大实现范围。

### 10.3 UI 体验

- UI 默认采用现代、清晰、响应式、可访问的设计风格；除非用户明确要求，不采用复古、实验性或强装饰风格。
- 页面信息架构必须符合目标用户习惯：核心任务优先、路径短、反馈清楚、状态可恢复。
- 禁止复杂功能堆叠、非人性化流程、隐藏关键操作、模糊按钮文案和不可解释的交互。

## 11. 版本、发布与提交管理

### 11.1 语义化版本

项目版本号采用 `MAJOR.MINOR.PATCH`：

- `MAJOR`：出现不兼容的公共 API、数据契约、配置契约或用户工作流变更。
- `MINOR`：向下兼容地新增功能、接口、配置项、文档章节或可选能力。
- `PATCH`：向下兼容的问题修复、措辞修正、示例修正或内部实现调整。

先行版本可使用 `MAJOR.MINOR.PATCH-prerelease`，例如 `1.4.0-beta.1`；构建元数据可使用 `+build` 后缀，例如 `1.4.0+20260601`。已发布版本的内容禁止原地修改，任何变更必须发布新版本。

### 11.2 文档版本

- 每次重大变更必须更新文档版本号
- 采用语义化版本：
  - MAJOR：规则体系重大调整
  - MINOR：新增章节或新约束
  - PATCH：措辞修正与小范围补充
- 在文末维护 Changelog

### 11.3 小步提交强约束

- 每个 commit 只能表达一个意图：功能、修复、重构、测试、文档、构建或 CI 中的一类。
- 每个 commit 必须能被独立解释、独立回滚，且不得破坏主干的构建和关键测试。
- 提交前必须执行与变更范围匹配的最小验证，并在提交说明或 PR 描述中记录验证命令与结果。
- 禁止提交“大包”：跨模块重构、格式化、依赖升级、功能开发混杂时，必须拆分。
- 禁止在未查看 `git diff`、未确认暂存内容时提交。
- 推荐提交信息格式：`type(scope): summary`，其中 `type` 可用 `feat`、`fix`、`docs`、`test`、`refactor`、`build`、`ci`、`chore`。
