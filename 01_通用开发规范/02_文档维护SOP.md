# 文档维护 SOP

版本：v2.1.0
状态：生效
最后更新：2026-06-27

## 1. 目标
规范文档更新流程，避免重复、冲突和版本漂移。

## 2. 维护范围
- 根目录总入口
- 01_通用开发规范
- 03_团队开发规范体系
- 02_个人开发者规范体系
- 04_设计参考资料

从属边界：
- `01_通用开发规范`：跨项目主规则、维护 SOP、大型项目和通用治理规则。
- `02_个人开发者规范体系`：独立开发者、小型项目和单人 AI 协作落地规则。
- `03_团队开发规范体系`：团队协作、评审、门禁、签署、发布治理和工程化模板。
- `04_设计参考资料`：网页视觉参考、品牌样例、外部 DESIGN.md 和原始参考资料。

内部层级：
- 个人版和团队版根目录只放 `00_总览...` 入口。
- `10_核心规范` 存放 PRD、流程、技术栈、前后端、实施、AI 协作、测试发布等开发主链路规范。
- `20_模板...` 存放追踪矩阵、初始化清单、术语、样例、门禁、评审和放行模板。
- `30_复盘与治理` 仅用于个人版复盘、成本治理和长期自我改进。
- `04_设计参考资料/90_外部原始资料` 只做原样归档，不参与规范主定义。

## 3. 更新流程

跨路径或跨类型规则的维护要求：

- 涉及 AI 执行、PR、Issue、测试门禁、发布、codegraph 或项目类型分流的变更，必须同步检查 `README.md`、`AGENTS.md`、`CLAUDE.md`、个人版入口、团队版入口和对应核心规范。
- 个人版和团队版可以保留场景差异，但不得降低通用规范中的安全、质量、提交和覆盖率红线。
- 纯文档规则修改必须执行目录自检、链接自检、版本与变更记录检查、`git diff` 审核和跨文档一致性检查。

## 3. 更新流程
1. 确定变更归属体系（通用规范、个人版、团队版、大型项目规范、设计参考或外部原始资料）
2. 确定所属层级（入口、核心规范、模板门禁、复盘治理、设计提炼、精选参考、外部原始资料）
3. 仅修改归属体系的主文档
4. 若影响入口导航、路径分流、项目类型分流或 AI 执行约束，再同步根目录 `README.md`、`AGENTS.md` 或 `CLAUDE.md`
5. 更新版本号与变更记录
6. 执行一次目录、链接、版本、变更记录和跨文档一致性自检
7. 若目录是 Git 仓库，按小步提交强约束提交：一类文档变更一个 commit，提交前查看 diff

## 3.1 Skill 与外部能力更新流程

当文档更新依据来自 GitHub star、Codex 支持 skill、插件、MCP 工具或外部 agent 框架时，必须额外执行：

1. 先区分来源类型：本地已支持能力、已 star 的候选能力、其他外部能力。
2. 只提炼可复用规则，不直接复制外部 README、prompt 或 skill 正文。
3. 检查能力是否涉及网络、写文件、命令执行、密钥读取或第三方数据传输。
4. 将通用准入规则写入 `01_通用开发规范`，将项目执行细节写入对应 `AGENT_RULES.md` 或 AI 协作文档。
5. 记录验证方式、停用条件和人工复核点。
6. 若能力改变开发、研究、审核或发布流程，必须同步检查 README 导航和相关变更记录。

## 4. 版本规则
- MAJOR：结构性调整（章节重构、门禁策略改版）
- MINOR：新增章节或新增模板
- PATCH：措辞修正、错字修复、示例优化

## 5. 命名规则
- 分文档采用 序号_主题_类型.md
- 序号保持两位数字（00-99）
- 同类文档仅允许一个主版本
- 根目录不新增散文件；通用规则放入 `01_通用开发规范`
- 根目录允许保留工具约定文件，如 `CLAUDE.md`，但只作为执行入口和规则索引
- 不得把团队工程规范放入设计参考目录；不得把外部设计素材放入 `03_团队开发规范体系`。
- 不得把核心规范放入模板目录；不得把模板提升为主规则来源。

## 6. 防重复规则
- 新建文档前先检查是否已有同主题文档
- 同一规则不得在多个文档中定义为主来源
- 跨文档内容复用优先用“引用说明”，不直接复制整段

## 7. 月度例行维护
- 检查失效链接与过时阈值
- 合并重复条目
- 更新门禁建议与成本模板
- 清理未使用模板

## 8. 验收清单
- 目录结构清晰
- 根入口可导航到通用规范、个人版、团队版、大型项目规范和设计参考
- 每份文档含版本、状态、更新时间
- 关键文档有变更记录
- `README.md` 中的路径均存在
- Markdown 内部不得残留已废弃目录路径
