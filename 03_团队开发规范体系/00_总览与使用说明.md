## 3. 阅读顺序

### 3.0 路径、类型与角色分流

团队项目开始前必须确认项目类型：Web 全栈、桌面/本地工具或 AI/Agent 工具，并追加对应检查项。

团队协作默认采用完整 RACI 和阶段角色结合：

- 需求阶段：产品/业务 Owner 负责范围，技术负责人评估可行性，测试负责人确认验收口径。
- 实现阶段：Executor 负责实现，Reviewer 负责代码与风险审核，Owner 确认不越界。
- 测试阶段：测试负责人或质量负责人确认核心路径、异常路径和回归证据。
- 发布阶段：发布负责人确认版本、回滚、监控和签署结论。

默认采用小阶段 PR：一个 PR 交付一段可验收用户路径、一个可独立审查的模块契约变化，或一个可独立验证的缺陷修复集合。

## 4. 基本原则

- 单一事实源：同一条规则只能在一个主文档中定义。
- 双向追踪：需求必须能追踪到实现、测试和发布记录。
- 量化验收：每个关键需求必须有可测量标准。
- 小步可回滚：每次变更都应最小化并可独立回退。
- 失败可复盘：所有异常需有证据链和修复结论。

## 5. 质量门禁总表

进入开发门禁（Gate-1）：
- PRD、APP_FLOW、TECH_STACK、IMPLEMENTATION_PLAN 均达到 B 级及以上。
- 关键需求均有 REQ 编号与验收标准。

进入测试门禁（Gate-2）：
- 追踪矩阵覆盖率达到 100%，即所有 REQ 均映射 TEST。
- P0 与 P1 功能均有自动化测试或明确人工验收脚本。
- 阶段代码审核问题已按标签提交 issue，并同步进入本地阶段报告。

进入发布门禁（Gate-3）：
- 发布前检查清单通过率 100%。
- 回滚脚本已演练并留存证据。
- 检查覆盖率达到 80% 以上。

## 6. 文档间引用关系

- PRD 定义业务目标与范围边界。
- APP_FLOW 定义用户路径与状态流转。
- TECH_STACK 定义实现边界与依赖约束。
- FRONTEND 与 BACKEND 分别定义表现层和数据层规则。
- IMPLEMENTATION_PLAN 绑定执行顺序与复现路径。
- 测试发布规范负责验收与收口。
