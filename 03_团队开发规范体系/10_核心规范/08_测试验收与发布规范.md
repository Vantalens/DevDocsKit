# 测试验收与发布规范

版本：v1.4.0
状态：生效
最后更新：2026-06-27

## 1. 目标
确保上线前质量可验证、风险可控、发布可回滚。

## 2. 测试分层

团队测试默认质量优先，同时按风险分级执行。80% 检查覆盖率是硬目标，核心用户路径、P0/P1、安全边界、异常路径和回滚路径优先于平均指标。构建通过不能替代用户路径通过。

- 单元测试：函数与组件级
- 集成测试：模块交互级
- 端到端测试：用户路径级
- 真实路径验证：按用户视角验证 Web、GUI、桌面或 Agent 核心路径，并保留截图、录屏、日志或命令输出证据
- 回归测试：历史缺陷防复发

强制覆盖率基线：
- 检查覆盖率大于等于 80%
- 单元测试覆盖率大于等于 80%
- 核心域模块覆盖率大于等于 85%
- 安全边界、异常路径、回滚路径和核心用户路径必须有明确验证证据

检查覆盖率口径包括自动化测试、静态检查、类型检查、lint、安全扫描、关键路径手工验证和阶段代码审核覆盖模块。

## 3. 测试用例规范
每个用例必须包含：
- 用例 ID：TEST-xxx
- 关联需求 ID
- 前置条件
- 测试步骤
- 预期结果
- 实际结果

## 4. 验收策略
### 4.1 功能验收
- 主路径通过率必须达到目标阈值
- 异常路径必须覆盖关键失败场景

功能验收建议阈值：
- P0 场景通过率 100%
- P1 场景通过率大于等于 95%

### 4.2 非功能验收
- 性能：接口延迟、页面关键指标
- 安全：越权、注入、敏感信息泄露检查
- 可用性：移动端与低网速行为

非功能验收建议阈值：
- 可用性：核心链路可用率大于等于 99%
- 性能：关键接口 P95 小于等于 500ms
- 安全：高危漏洞为 0

## 5. 发布前检查清单
- 环境变量已配置且无敏感泄露
- 数据迁移脚本已演练
- 回滚方案已验证
- 监控告警已生效
- 阶段代码审核已完成
- 审核问题已按标签提交 issue 并同步本地阶段报告，P0/P1 与安全阻断项已关闭或豁免
- codegraph 或等价工具已梳理本次变更影响面

## 6. 发布与回滚流程
- 灰度发布优先
- 观察窗口内跟踪核心指标
- 触发阈值后自动或手动回滚

## 6.1 发布分级

发布前必须确认本次变更对应的小阶段 PR 边界、RACI 签署、核心路径验证证据和本地阶段报告。非阻断评审问题不得混入发布 PR，必须通过 issue 和后续单独 PR 处理。

- L1（低风险）：仅文案或样式微调
- L2（中风险）：单模块逻辑改动，无 schema 变化
- L3（高风险）：跨模块改动、数据迁移、鉴权链路改动

要求：
- L3 必须进行演练发布并由技术负责人签署
- L2/L3 必须具备可执行回滚脚本

## 6.2 发布阻断条件
满足任一条件即阻断发布：
- 存在未关闭的 P0 缺陷
- 核心流程端到端用例未通过
- 回滚方案未演练或证据缺失
- 监控告警未配置或未生效
- 本次变更未按小步提交拆分，或提交缺少验证证据
- 检查覆盖率低于 80%
- 阶段代码审核未执行，或审核问题未登记 issue 与本地阶段报告
- 安全、高风险数据、核心流程问题未关闭或未豁免

## 6.3 版本号规则
- 发布版本采用语义化版本 `MAJOR.MINOR.PATCH`。
- 不兼容公共 API、数据契约、配置契约或用户工作流变更，升级 `MAJOR`。
- 向下兼容的新功能、新接口、新配置项或弃用声明，升级 `MINOR`。
- 向下兼容的问题修复、文档修正或内部实现调整，升级 `PATCH`。
- 先行版本使用 `-alpha`、`-beta`、`-rc` 等后缀，构建信息使用 `+build` 后缀。
- 已发布版本不得原地改内容，任何修正必须形成新版本。

## 6.4 小步提交强约束
- 一个 commit 只允许包含一个可说明、可验证、可回滚的意图。
- 每个需求、缺陷或重构任务必须先拆为原子任务，再映射到一个或多个 commit。
- 提交前必须查看 diff、确认暂存范围、执行与变更范围匹配的最小验证。
- 禁止把功能、修复、重构、格式化、依赖升级、测试补齐混在同一个 commit。
- 提交信息推荐使用 `type(scope): summary`，PR 或发布记录必须包含验证命令与结果。

## 7. 发布后复盘
- 记录发布版本、时间、影响范围
- 记录异常与处理时长
- 更新 lessons 与规范文档
- 汇总本阶段代码审核发现的问题、issue 标签分布、本地阶段报告位置、关闭状态和遗留风险

复盘模板建议新增：
- 触发事件时间线
- 发现到恢复时长（MTTR）
- 防复发动作与截止日期

## 9. 评审清单
- 用例是否覆盖需求与流程
- 发布是否具备完整回滚路径
- 是否有发布后复盘结论
- 检查覆盖率是否大于等于 80%
- 阶段代码审核是否覆盖正确性、健壮性、鲁棒性、安全性、性能和用户体验
- 审核问题是否已按标签提交 issue 并同步本地阶段报告
- 是否使用 codegraph 梳理影响面
