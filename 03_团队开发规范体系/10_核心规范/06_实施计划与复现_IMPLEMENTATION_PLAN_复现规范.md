# 实施计划与复现规范 IMPLEMENTATION_PLAN

版本：v1.2.0  
状态：生效  
最后更新：2026-06-09

## 1. 目标
把开发工作拆解为可执行、可验证、可复现的最小任务单元。

## 2. 任务拆分规则
- 每个任务应在 0.5 到 1 人天内闭环
- 每个任务必须有输入与输出定义
- 每个任务必须绑定需求编号和测试编号
- 每个任务必须指定负责人与评审人
- 每个任务必须给出完成定义（DoD）
- 每个任务必须绑定一个主模块；跨模块任务必须说明模块契约变化、影响范围和回滚方式
- 每个阶段完成后必须进入代码审核，不得直接跳到下一阶段
- AI 不得提前实现未来阶段或未进入当前计划的复杂功能；可选优化登记为 issue

## 3. 任务条目强制字段
- Task ID
- 对应需求 ID
- 前置条件
- 操作步骤
- 预期产物
- 验收方式
- 回滚步骤
- 负责人与评审人
- 关联代码范围
- 变更影响分析（功能、数据、性能、安全）
- 关联模块与模块边界
- codegraph 影响面梳理结论
- 阶段代码审核方式
- issue 标签化报告要求
- 覆盖率目标与检查命令

## 4. 复现规范
### 4.1 环境复现
- 系统信息
- 依赖版本
- 环境变量清单（仅变量名）
- 启动命令与顺序

### 4.2 问题复现
必须记录：
- 触发步骤
- 预期结果
- 实际结果
- 日志证据
- 影响范围
- 首次出现版本与最近正常版本
- 复现概率与触发条件

### 4.3 修复复现
必须记录：
- 根因定位
- 修复动作
- 回归验证
- 未覆盖风险
- 证据链接（日志片段、测试结果、截图/录屏）

### 4.4 复现成功判定
- 在干净环境连续执行 2 次结果一致
- 关键步骤无人工口头补充即可完成
- 复现文档可被非作者在 30 分钟内复现成功

## 5. 追踪矩阵
建立矩阵：REQ -> FLOW -> TASK -> API/COMPONENT -> TEST -> RELEASE NOTE

扩展矩阵必须包含：MODULE -> CODEGRAPH_QUERY -> REVIEW -> ISSUE -> COVERAGE。

## 6. 分支与提交规范
- 分支命名建议：feature/xxx、fix/xxx、refactor/xxx
- 提交信息建议：类型(范围): 摘要
- 每个提交需可独立解释其价值

## 6.1 任务状态机
状态流转建议：
- todo -> in_progress -> in_review -> done
- todo/in_progress -> blocked
- blocked -> in_progress（解除阻塞后）

约束：
- 同一任务同一时刻只能有一个 in_progress 负责人
- 进入 in_review 前必须附带复现与验证证据
- 进入 in_review 前必须完成 codegraph 或等价索引影响面梳理
- 从 in_review 到 done 前必须完成代码审核、issue 登记和覆盖率检查
- 若覆盖率低于 80%，任务不得进入 done，只能进入 blocked 或继续修复

## 6.2 阶段代码审核与 Issue 规则

每个阶段结束后必须执行代码审核，覆盖正确性、健壮性、鲁棒性、安全性、性能、可维护性、可测试性和用户体验。

审核问题必须提交 issue 或等价跟踪项，至少包含：
- 标题
- 问题描述
- 影响模块
- 复现或证据
- 建议修复方向
- 标签：严重级别、类型、模块、阶段

P0/P1、安全、核心流程、数据一致性和覆盖率阻断问题必须关闭或豁免后，阶段才能完成。

## 8. 评审清单
- 是否能在新环境完整复现
- 是否有回滚路径
- 是否已更新追踪矩阵
- 是否补充了证据链
- 是否完成影响分析
- 是否按模块化方式拆分任务
- 是否使用 codegraph 梳理代码影响面
- 是否完成阶段代码审核并登记 issue
- 是否检查覆盖率大于等于 80%
- 是否避免 AI 过度超前实现
