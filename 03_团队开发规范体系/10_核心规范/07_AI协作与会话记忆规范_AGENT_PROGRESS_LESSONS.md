# AI 协作与会话记忆规范 AGENT_PROGRESS_LESSONS

版本：v1.3.0
状态：生效
最后更新：2026-06-27

## 1. 目标
让 AI 在多会话、多阶段开发中保持上下文连续、规则稳定、错误可学习。

## 2. 文件职责

团队项目采用三层记忆和责任记录：progress 记录当前状态，lessons 记录可复用错误模式，阶段归档记录证据链、RACI、审核结论、issue/PR 链接和发布结论。

文件职责：
- AGENT_RULES：行为约束与执行顺序
- progress：当前进度与阻塞状态
- lessons：错误模式与纠正策略
- 阶段归档：阶段目标、RACI、验证证据、审核结论、issue/PR 链接和收尾报告

## 3. 会话启动协议

AI 默认作为执行代理：按已冻结文档、RACI 和实施计划连续推进普通任务。遇到安全、数据迁移、删除、依赖升级、架构边界、生产发布、关键代码提交等高风险动作，必须停下确认。

每次会话必须先完成：
1. 阅读 AGENT_RULES
2. 阅读 progress
3. 阅读与当前任务相关的规范文档
4. 检查当前任务是否匹配 Codex 当前支持的 skill、插件或 MCP 工具
5. 阅读项目开发文档：`TECH_STACK.md`、`IMPLEMENTATION_PLAN.md`、`TEST_PLAN.md`、`AGENT_RULES.md` 或等价文件
6. 使用 codegraph 或等价本地索引梳理目标模块、调用关系和影响面
7. 输出任务边界、项目类型、模块范围、RACI/阶段角色、输入、输出、验收、可用 skill、权限边界和代码审核计划

## 4. 会话执行协议
- 一次只执行一个任务目标
- 每次重大决策需记录决策理由
- 发现文档冲突时先修正文档再改代码
- 优先使用本地已支持 skill；GitHub star 中的开发或研究 skill 只能作为候选能力和方法论来源
- 引入外部 skill、MCP 工具或代理插件前，必须完成来源、许可、权限、输入输出、验证方式和停用条件记录
- 对代码理解、研究、代码审核、数据分析、部署和运维类任务，必须说明选择该 skill 的理由以及未选择替代方案的理由
- AI 不得过度超前，不得实现当前需求、实施计划或用户明确要求之外的复杂功能
- 开发必须按模块化方式推进，跨模块修改必须先说明契约变化和影响面
- P0/P1、安全或阻断当前用户路径的问题在当前 PR 内修复；非阻断问题进入 issue 和本地阶段报告，后续单独 PR 处理

## 5. 会话收尾协议
- 更新 progress：Done/In Progress/Next/Blockers
- 记录已知问题与下一步建议
- 更新阶段归档：RACI、验证证据、审核结论、issue/PR 链接和未覆盖风险
- 将新错误模式写入 lessons
- 记录本轮 skill 使用结果：有效场景、失败场景、证据、是否继续保留
- 阶段完成后执行代码审核，并把问题按标签提交 issue 或给出可提交 issue 内容，同时同步本地阶段报告
- 报告检查覆盖率是否大于等于 80%，未达标时记录阻断原因
- 记录 codegraph 梳理结论和未覆盖风险

## 6. AI 输出质量约束
- 必须引用依据文档
- 必须给出可验证变更点
- 禁止无依据新增依赖与越界改动
- 必须区分外部资料结论、工具输出结论和项目内已验证事实
- 不得把 GitHub star、流行度或工具自述当作充分采纳理由
- 必须覆盖正确性、健壮性、鲁棒性、安全性、性能、可维护性、可测试性和用户体验的审核结论，并说明阻断项是否已在当前 PR 修复
- UI 默认采用现代、清晰、可访问、响应式风格，页面内容必须符合用户使用习惯

## 8. 评审清单
- 是否按启动协议执行
- 是否按收尾协议更新
- 是否沉淀可复用经验
- 是否优先检查 Codex 已支持 skill
- 是否记录外部 skill 的权限边界、验证方式和停用条件
- 是否阅读项目开发文档并使用 codegraph 梳理影响面
- 是否完成阶段代码审核和 issue 标签化报告
- 是否达到 80% 检查覆盖率
- 是否避免 AI 过度超前和复杂化 UI
