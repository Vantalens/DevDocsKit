## 1. 定位

本文件是 Codex、Claude Code 或兼容 AI 代理在 DevDocsKit 中执行任务时的项目级入口。它不替代 `01_通用开发规范`、`02_个人开发者规范体系` 或 `03_团队开发规范体系`，只负责把执行顺序、提交约束和验证红线明确化。

## 2. 启动顺序

每次开始任务前必须先阅读：

1. `README.md`
2. `01_通用开发规范/02_文档维护SOP.md`
3. `01_通用开发规范/01_开发标准文档编写规则.md`
4. `01_通用开发规范/03_大型项目开发规范.md`（长期维护、多模块或高风险项目必须读）
5. 与任务场景匹配的 `02_个人开发者规范体系` 或 `03_团队开发规范体系` 入口和核心规范
6. 项目内已有的 `AGENT_RULES.md`、`progress.md`、`lessons.md`、`TECH_STACK.md`、`IMPLEMENTATION_PLAN.md`、`TEST_PLAN.md` 或等价开发文档（如果存在）

读取后先说明本次任务边界、模块范围、预计修改文件、验证方式和代码审核方式。若用户明确要求直接执行，可以用一句话说明后继续，不做冗长计划。

## 2.1 开发路径与项目类型分流

DevDocsKit 默认同时支持两条主路径：

- 个人开发路径：适用于个人本地全栈、个人产品 MVP、长期维护的个人项目。
- 团队开发路径：适用于多人协作、PR 合并、RACI 分工、阶段评审和发布签署。

每次进入项目或阶段前，必须先选择项目类型，并追加对应检查项：

- Web 全栈项目：重点确认前端、后端、数据库、鉴权、部署、端到端用户路径和回滚。
- 桌面/本地工具项目：重点确认 GUI 入口、本地运行时、文件读写、打包、安装、离线复现和用户数据边界。
- AI/Agent 工具项目：重点确认模型调用、工具权限、任务编排、记忆、成本、可审计输出和外部服务边界。

进入开发前先冻结关键闭环：目标、范围、流程、技术栈、实施阶段、测试验收、回滚和 AI 执行边界。完整规格允许随小阶段开发持续补充，但每个阶段开始前必须重新冻结本阶段范围和验收标准。

## 3. Codegraph 与代码库梳理

- 项目库开始开发前必须优先使用 codegraph 或等价本地代码索引工具梳理模块、调用关系、影响面和风险点。
- codegraph 输出用于减少盲目全文搜索、降低 token 消耗、提升代码库管理和理解效率。
- codegraph 结论不能替代测试、构建、代码审核或人工验收；涉及修改时仍需用最小验证闭环。
- 若项目尚未初始化 codegraph，必须在任务说明中记录未初始化原因，并在可行时建议建立索引。
- 纯文档微调、错字修正、导航修正等低风险任务可使用简化影响面检查；必须说明未使用 codegraph 的原因和替代检查方式。
- 纯文档规范库任务可用目录自检、链接自检、版本与变更记录检查、`git diff` 和跨文档一致性检查替代代码调用分析。

## 4. 模块化开发约束

- 开发必须按模块边界推进：先确认模块职责、公开接口、依赖方向和禁止访问范围，再修改代码。
- 禁止跨模块无计划扩散、顺手重构和把多个不相关模块合并在同一任务中修改。
- 每个阶段必须说明本阶段影响的模块、未触碰模块和跨模块契约变化。
- 公共模块、基础设施模块和安全相关模块变更必须提高审核和测试等级。

## 5. 小步提交强约束

- 一个 commit 只允许包含一个可说明、可验证、可回滚的意图。
- 每个原子任务完成后，必须先运行匹配范围的最小验证，再提交。
- 每个阶段开发完成后，必须先执行代码审核并处理或登记问题，再进入下一阶段或发布流程。
- 提交前必须查看 `git status` 和 `git diff`，确认暂存内容只属于本次任务。
- 禁止把功能、修复、重构、格式化、依赖升级、测试补齐混在同一个 commit。
- 禁止为了“提交少一点”合并无关改动。
- 默认采用小阶段 PR：一个 PR 交付一段可验收用户路径或一个可独立审查的文档阶段。
- 关键代码提交前必须询问确认；文档提交按任务约定执行，未获确认不得擅自 commit。
- 若当前目录不是 Git 仓库，必须明确说明无法执行 commit，只能完成文件更新与人工检查。

推荐提交信息：

```text
type(scope): summary
```

常用 `type`：`feat`、`fix`、`docs`、`test`、`refactor`、`build`、`ci`、`chore`。

## 6. 阶段代码审核与 Issue 报告

- 每个开发阶段结束后必须执行全面代码审核，覆盖正确性、健壮性、鲁棒性、安全性、性能、可维护性、可测试性和用户体验。
- 审核发现的问题不得只写在聊天记录中；必须按问题创建 issue 或等价跟踪项，并同步写入本地阶段报告。
- issue 必须带标签，最低标签集包括：严重级别（`P0/P1/P2/P3`）、类型（`bug/security/performance/robustness/ux/test/docs/refactor`）、模块名、阶段名。
- 本地阶段报告必须记录问题摘要、证据、影响范围、修复建议、状态、对应 issue 或 PR 链接。
- 无法立即创建 issue 时，必须在收尾报告中给出可直接提交的 issue 标题、正文、标签和复现/验证证据。
- P0/P1、安全、高风险数据、核心流程和覆盖率阻断问题必须在合并或发布前关闭或由负责人书面豁免。
- P0/P1、安全或阻断当前用户路径的问题在当前 PR 内修复；非阻断问题 issue 化后单独 PR 处理。

## 7. 质量、测试与覆盖率红线

- 检查覆盖率强制要求大于等于 80%，包括自动化测试、静态检查、类型检查、lint、安全扫描和关键路径验证的综合门禁。
- P0 用户路径、核心模块、安全边界、异常路径和回滚路径必须纳入审核和验证范围。
- 代码必须具备良好的健壮性、鲁棒性、安全性和高性能；不得只验证成功路径。
- 覆盖率或检查项未达标时，不得宣称阶段完成，只能说明未达标项、风险、补救计划和阻断状态。
- 成长型门禁表中的单元测试覆盖率阈值只用于成熟度爬坡，不降低发布前检查覆盖率 80% 的硬门槛；如需豁免，必须记录适用范围、负责人和补救计划。

## 8. AI 开发节制与 UI 体验

- AI 默认作为执行代理：按冻结文档、实施计划和用户任务连续推进，普通实现不反复打断用户。
- 安全、数据迁移、删除、依赖升级、架构边界、生产发布、关键代码提交等高风险动作必须先停下确认。

- AI 开发不得过度超前：不得在当前 PRD、APP_FLOW、IMPLEMENTATION_PLAN 或用户明确要求之外扩展复杂功能。
- 发现可选优化时，优先登记为后续 issue，不直接混入当前阶段。
- UI 界面默认采用现代、清晰、可访问、响应式风格；除非用户明确要求，不采用复古、拟物、实验性或强装饰风格。
- 页面内容必须符合目标用户的使用习惯，优先清晰流程、常见控件、明确反馈和低认知负担。
- 禁止复杂功能堆叠、隐藏关键路径、非人性化交互、无必要的多步操作或只为展示效果牺牲可用性。

## 9. 版本规则

发布版本采用语义化版本 `MAJOR.MINOR.PATCH`：

- `MAJOR`：不兼容的公共 API、数据契约、配置契约、模块边界或用户工作流变更。
- `MINOR`：向下兼容的新功能、新接口、新配置项、弃用声明或文档章节。
- `PATCH`：向下兼容的问题修复、文档修正、示例优化或内部实现调整。

已发布版本不得原地修改；任何修正都必须形成新版本。先行版本使用 `-alpha`、`-beta`、`-rc`，构建信息使用 `+build`。

## 10. 文档更新规则

- 通用规则优先写入 `01_通用开发规范`，个人版和团队版只做场景化落地或引用。
- 团队工程化开发规则归入 `03_团队开发规范体系`；网页视觉参考、品牌样例和外部设计资料归入 `04_设计参考资料`。
- `02_个人开发者规范体系` 与 `03_团队开发规范体系` 内部必须区分入口、核心规范、模板门禁和复盘治理，不再把所有文档平铺在同一层。
- 根目录只保留入口文件和工具约定文件，新增规范不得散落在根目录。
- 修改文档时必须同步更新版本、最后更新时间和变更记录。
- 调整入口、目录或跨文档规则时，必须同步检查 `README.md`。

## 11. 执行红线

- 不覆盖用户未要求修改的内容。
- 不删除未确认用途的文件、配置、脚本或数据。
- 不把未验证的猜测写成完成结论。
- 不在文档中记录真实密钥、令牌、密码或生产连接串。
- 不引入未记录版本和用途的新依赖。
- 不跳过阶段代码审核、issue 报告、覆盖率门禁或 codegraph 梳理。

## 12. 收尾要求

完成任务后必须说明：

- 修改了哪些文件。
- 执行了哪些验证。
- 代码审核结果、issue 报告方式、标签和剩余问题状态。
- 覆盖率或检查覆盖是否达到 80% 以上；未达到时说明阻断原因。
- 是否使用 codegraph 梳理项目库；如果未使用，说明原因。
- 是否完成 commit；如果不能 commit，说明原因。
- 剩余风险或需要人工确认的事项。
